Property Operations API
with a Bit of AI

Build a small API that uses an AI model to turn plain language maintenance requests into clean, structured work orders.

Stack  PHP / Laravel Focus  API design and working with AI

About this test

Thanks for taking the time to do this. At Proprli we help property teams look after their buildings and keep track of the work that needs doing. More and more of that involves AI, helping people turn a few words written in plain language into something clean and useful.

So that is what we would like you to build here: a small API that has an AI model doing real work inside it. It is about a made up property portfolio, so you do not need any background in real estate. There is no front end to build. We mostly want to see how you design an API, how you work with an AI model as part of it, and how tidy your code is.

What we want you to build

A REST API built with PHP and Laravel. It works with two simple things:

  • Buildings, which are the properties in the portfolio.
  • Work orders, which are jobs that need doing in a building, like fixing an elevator or replacing a filter.

The fun part is how work orders get created. Instead of filling in a tidy form, someone just describes the problem in plain words, the way a tenant or a caretaker actually would. Your API sends that text to an AI model, and the model helps turn it into a proper work order with a clear title, a category, a priority, and a short summary. Your API then saves that and hands it back.

The data you will work with

You can keep the buildings in a couple of seed files or a small database, whichever you prefer. Here is the kind of data we mean.

// a building
{
  "id": "P-001",
  "name": "Keizersgracht 128",
  "type": "office",
  "status": "active",
  "city": "Amsterdam",
  "units": 14,
  "occupancy_rate": 0.86,
  "amenities": ["elevator", "parking"]
}
// a work order, after the AI has helped fill it in
{
  "id": "WO-1001",
  "property_id": "P-001",
  "source_text": "the elevator in the lobby keeps stopping and makes a grinding noise",
  "title": "Lobby elevator stopping and making noise",      // from the AI
  "category": "elevator",                                 // from the AI
  "priority": "high",                                    // from the AI
  "summary": "Lobby elevator is stopping between floors...",   // from the AI
  "status": "open",
  "created_at": "2025-05-02"
}

Please include at least twelve buildings with a bit of variety, so different cities, types and occupancy levels, and leave a few fields empty here and there so your code has to cope with data that is not always complete.

What the API should do

There are four things the API needs to handle. The paths below are just a suggestion, so shape the routes however feels natural to you.

1GET/api/properties

Return a list of buildings, with the option to filter by things like city, type, status, or a minimum occupancy. Show the fullest buildings first. If nothing matches, say so clearly instead of returning an empty list with no explanation.

2GET/api/properties/{id}

Return the full details of one building, along with how many open work orders it has. If the building does not exist, return a clear "not found" response.

3POST/api/work-ordersUSES AI

This is the heart of the test. Someone sends the id of a building, their email, and a plain language description of the problem. Your API passes that description to an AI model, which comes back with a clean title, a category, a priority, and a short summary. You then save a new work order that keeps the original text and adds those fields, with a status of "open" and the date it was created, and return it. Before saving, check that the building exists and that the email looks real.

4GET/api/work-orders

Return the work orders, with the option to filter by building, status, priority, or category. Show the most urgent and most recent ones first.

Working with the AI model

Which model to use

Use whichever AI provider you like, such as OpenAI or Anthropic. Keep the key in an environment variable and include a .env.example so we know what to set, but never commit your real key.

You do not need to spend anything (if you don't want)

There is no need to pay for this. Several models have a free tier that is more than enough here, with some limits on how often you can call them. Options like Google Gemini, Groq and a few of the free models on OpenRouter all work well, so pick whichever you like. And if you would rather use a paid model you are comfortable with, that is completely fine too. Just say which one you went with in your README.

A few things we care about

  • Keep the AI call tucked away behind its own small piece of code, so it is easy to read and swap out later.
  • Check the model's answer before you trust it, and never save a half filled work order because the model slipped up.
  • Keep the rest of the code tidy too, with the data logic kept apart from the routes and controllers.
  • Use sensible HTTP status codes, and give helpful responses when something goes wrong.

These are all optional. A clean, working version of the main API means much more to us than a pile of half finished extras.

  • Please do not over build it. We care as much about your judgment as about how much you get done.
